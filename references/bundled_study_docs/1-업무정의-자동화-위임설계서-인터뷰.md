## 📌 이 가이드의 목적

> **"AI팀이 완벽하게 업무를 마쳤을 때의 장면"을 먼저 그리고,**
> **그 장면을 만들기 위해 필요한 업무를 단계별로 정의합니다.**

이 가이드는 인터뷰 형식으로 질문에 답하면서 업무를 구조화하고, AI팀에게 위임할 수 있는 수준으로 정의하는 실전 프레임워크입니다.

**질문은 반드시 한 번에 1개의 질문만 해야 합니다. 빠짐없이 모든 질문을 진행해야 합니다. 완벽한 인터뷰가 되도록 질문과 답변을 모두 정리하여 워크시트 산출물을 완성해야 합니다. 사용자가 현재 더이상 답할 수 없는 항목만 `[보류: 이유]`로 남기고 넘어갈 수 있습니다.**

## Step 0️⃣ 시작 전 확인

> **이 업무의 자동화는 누구를 위한 것인가요?**

| 질문 | 선택 |
|------|------|
| 자동화 대상 | □ **나 혼자의 업무** (권장) / □ 여러 사람의 업무 |

> 💡 **권장**: 처음에는 **나 1명의 업무**를 자동화하는 것부터 시작하세요.
> 다수의 수혜자를 위한 자동화는 업무 정의가 매우 복잡해지므로, 1인 자동화를 먼저 완성한 후 확장하는 것이 현실적입니다.

## Step 1️⃣ 참여자·수혜자 매핑 (Stakeholder Map)

> **"이 업무에 관여하는 모든 사람과 각각의 역할은 무엇인가?"**

### 1-A. 업무 참여자 전체 식별

업무에 관련된 모든 사람(역할)을 빠짐없이 나열합니다.

| 질문 | 설명 |
|------|------|
| 이 업무를 **시작시키는 사람**(요청자)은 누구인가? | 예: 팀장, 고객사, 나 자신 |
| 이 업무를 **직접 수행하는 사람**(실행자)은 누구인가? | 예: 나, 팀원, AI |
| 이 업무의 **결과물을 받는 사람**(수혜자)은 누구인가? | 예: 나, 팀장, 고객사 담당자 |
| 이 업무의 **중간 확인/승인이 필요한 사람**(의사결정자)은? | 예: 팀장, 법무팀, 내가 직접 확인 |
| 업무 중 **다른 사람에게 넘겼다가 돌려받는 구간**이 있는가? | 예: 초안 → 팀장 검토 → 수정본 돌려받음 |
| 그 밖에 **참고하거나 통보받아야 할 사람**이 있는가? | 예: CC 대상, 보고 라인 |

### 1-B. 참여자 간 업무 흐름 (현재 상태)

참여자 간에 **현재** 어떤 업무를 주고받는지 그립니다.

```
[요청자] ──요청──→ [실행자(나)] ──결과물──→ [수혜자]
                      ↑                      ↓
                 [의사결정자] ←──확인요청──  [수혜자 피드백]
```

| From | To | 주고받는 것 | 빈도 | 형태 |
|------|----|-----------|------|------|
| (예: 팀장) | (예: 나) | 보고서 작성 요청 | 매주 월 | 구두/메신저 |
| (예: 나) | (예: 팀장) | 완성된 주간보고 | 매주 금 | 이메일+파일 |
| | | | | |
| | | | | |

### 1-B-2. 협업 연결 흐름 (Handoff & Handback)

> 💡 업무가 **다른 참여자에게 넘어갔다가 다시 돌아오는 경우**가 있습니다.
> 이런 경우 자동화는 "완전 종료"가 아니라 **"일시 구간 종료"** 후, 사용자의 재개 명령으로 이어지는 구조입니다.

업무 중 **다른 참여자에게 넘기는 구간**이 있는지 확인합니다.

```
[AI 자동화 구간 A] ──핸드오프──→ [사람/외부 협업 구간] ──핸드백──→ [AI 자동화 구간 B]
       🟢                            🔴/🟡                           🟢
   (일시 구간 종료)              (사람이 처리)                  (재개 명령 후 자동화 재개)
```

| 질문 | 설명 |
|------|------|
| 업무 중 **다른 사람에게 넘겨야 하는 구간**이 있는가? | 예: 초안 → 팀장 검토 → 수정 반영 |
| 넘긴 후 **돌아오는 결과물**은 무엇인가? | 예: 승인된 초안, 수정 코멘트, 추가 데이터 |
| 넘기고 돌아오는 **대기 시간**은 얼마나 되나? | 예: 1시간, 당일 중, 2~3일 |
| 돌아온 결과물로 **AI가 다시 이어서 할 작업**은? | 예: 수정 반영 후 최종본 생성 |

**협업 연결 흐름표:**

| # | AI 구간 (자동화) | 핸드오프 대상 | 넘기는 것 | 돌아오는 것 | 대기 시간 | AI 재개 작업 |
|---|---------------|------------|---------|----------|---------|------------|
| (예) | 보고서 초안 생성 | 팀장 | 초안 PDF | 수정 코멘트 | 당일 중 | 코멘트 반영 후 최종본 생성 |
| 1 | | | | | | |
| 2 | | | | | | |

> ⚠️ **핵심**: 핸드오프 → 핸드백이 있으면 자동화는 **한 번에 끝나지 않습니다.**
> 이 구간을 명확히 정의해야 워크플로우 지시서에서 "일시 중단 → 대기 → 재개 명령" 로직을 설계할 수 있습니다.

### 1-C. 수혜자별 상세 분석

> 디폴트는 **1명**(나 자신)입니다. 수혜자가 복수인 경우에만 추가 작성합니다.

**수혜자 #1 (필수):**

| 항목 | 내용 |
|------|------|
| 수혜자 (이름/역할) | |
| 현재 상황 | |
| 이 업무에서 기대하는 것 | |
| 가장 중요하게 여기는 품질 기준 | |
| 결과물을 받는 채널/형식 | |

**수혜자 #2:** (해당 시 추가)

| 항목 | 내용 |
|------|------|
| 수혜자 (이름/역할) | |
| 현재 상황 | |
| 이 업무에서 기대하는 것 | |
| 가장 중요하게 여기는 품질 기준 | |
| 결과물을 받는 채널/형식 | |

### 작성 포인트
- 🎯 수혜자 = 나(스스로) 여도 완전히 OK. **초기에는 1명을 권장**
- 수혜자가 여럿이면 **모두** 나열하되 **1순위**를 선정
- "팀 전체"처럼 광범위하면 → 구체적 역할로 분해 (예: 팀장, 실무자 A, 고객사 B)
- 참여자 간 현재 커뮤니케이션 채널(이메일, 메신저, 구두 등)도 기록
- 🔄 업무가 **다른 참여자에게 넘어갔다가 돌아오는 구간**(핸드오프/핸드백)이 있으면 반드시 1-B-2에서 정의. 이 구간이 누락되면 자동화가 "완전 종료"로 설계되어 이어지는 업무가 끊어짐

## Step 2️⃣ 현재 고통 정의 (Problem) — 수혜자별

### 작성 포인트
- ❌ "AI로 자동화하고 싶다"는 해결책이지 **문제가 아님** → 문제를 써야 함
- 시간 순서로 나열하면 **병목**이 보이고, 수혜자별로 쓰면 **누구의 문제인지** 명확해짐
- 불만스럽지 않더라도 **시간을 가장 많이 쓰는 업무**도 자동화 대상이 됨
- **실수/누락 패턴**을 기록하면 → 예외정책(Step 5)의 재료가 됨

## Step 3️⃣ 위임 선언서 (Delegation Announcement)

> **"AI팀이 이 업무를 완수했을 때의 장면을 지금 쓴다"**

### 3-A. 자동화 범위 선언

이 업무의 **어디부터 어디까지**를 자동화할지 명확히 구분합니다.

| 단계 | 현재 수행자 | 자동화 유형 | 설명 |
|------|-----------|-----------|------|
| (예: 데이터 수집) | 나 | 🟢 **완전 자동화** | AI가 100% 처리, 사람 개입 불필요 |
| (예: 요약 작성) | 나 | 🟡 **부분 자동화** | AI가 초안 작성, 사람이 검토/수정 |
| (예: 최종 승인) | 팀장 | 🔴 **사람 전용** | AI 개입 불가, 사람이 반드시 판단 |
| | | | |

### 자동화 유형 정의

```
🟢 완전 자동화:       AI가 시작부터 끝까지 처리. 사람은 결과만 확인.
🟡 부분 자동화:       AI가 실행하되, 특정 지점에서 사람의 확인/검증/수정 필요.
🟡⏸️ 일시 구간 종료:  AI가 중간 산출물까지 처리 후 일시 중단.
                      사람/외부 협업 완료 후 재개 명령을 받으면 AI가 이어서 처리.
🔴 사람 전용:         반드시 사람이 직접 판단/실행. AI는 보조 정보만 제공.
```

> 💡 **🟡⏸️ 일시 구간 종료**는 부분 자동화의 특수 형태입니다.
> 일반 🟡는 "사람이 확인하면 바로 다음 단계로 진행"이지만,
> 🟡⏸️는 "사람/외부 협업이 끝날 때까지 대기했다가, 결과물과 재개 명령을 받아 AI가 다시 이어서 처리"합니다.

### 3-B. 사람 개입 지점 (Human-in-the-Loop)

🟡 부분 자동화 단계에서 **사람이 무엇을 확인/결정하는지** 구체화합니다.

| 단계 | 사람이 확인할 내용 | 판단 기준 | 확인 후 행동 |
|------|-----------------|---------|------------|
| (예: 요약 검토) | 요약 내용의 정확성 | 사실 오류 없음 | 승인 → 다음 단계 진행 |
| (예: 발송 전 확인) | 수신자·첨부 확인 | 민감 정보 미포함 | 승인 → 자동 발송 |
| | | | |

### 3-B-2. 협업 연결 지점 (Handoff & Handback)

🟡⏸️ 일시 구간 종료가 있는 단계에서 **AI가 멈추는 시점, 사람/외부가 처리하는 내용, AI가 다시 이어받는 명령 조건**을 구체화합니다.

```
[AI 구간 A: 자동 처리] ──핸드오프──→ [사람/외부 구간] ──핸드백──→ [AI 구간 B: 재개 명령 후 처리]
     🟢 or 🟡                         🔴 or 🟡                      🟢 or 🟡
  AI가 중간 산출물 생성           사람이 검토/수정/승인            AI가 결과물 받아 이어서 처리
  → 지정 위치에 저장             → 지정 위치에 결과 저장          → 재개 명령 수신 후 이어서 처리
```

| # | AI 멈추는 시점 | 중간 산출물 | 핸드오프 대상 | 사람/외부 처리 내용 | 핸드백 결과물 | AI 재개 명령 | AI 재개 작업 |
|---|-------------|----------|------------|----------------|----------|-------------|------------|
| (예) | 보고서 초안 완성 | 초안.md | 팀장 | 내용 검토 + 코멘트 | 수정코멘트.md | "초안 수정 반영해줘" | 코멘트 반영 → 최종본 생성 |
| 1 | | | | | | | |
| 2 | | | | | | | |

**재개 명령 유형:**

| 명령 유형 | 설명 | 예시 |
|-----------|------|------|

## Step 6️⃣ AI 검증 및 보완 질문

> **"워크시트 작성이 끝나면 AI가 검증하고 보완 질문을 합니다"**

### 6-A. 자가 점검 체크리스트 검증

AI는 아래 체크리스트 항목을 **하나씩 검증**합니다.

```text
□ 참여자(요청자/실행자/수혜자/의사결정자)가 빠짐없이 식별되었는가?
□ 참여자 간 현재 업무 흐름(주고받는 것, 빈도, 형태)이 기록되었는가?
□ 수혜자별로 고통 분석이 각각 작성되었는가?
□ 자동화 범위가 🟢/🟡/🔴로 구분되었는가?
□ 🟡 부분 자동화에서 사람 개입 지점이 구체적으로 정의되었는가?
□ 대표 실행 명령과 명령 채널이 명확히 정의되었는가?
□ 입력 데이터의 소스/형태/항목이 정의되었는가? (템플릿은 별도 첨부 가능)
□ 출력 결과물의 형태/파일명/저장위치가 정의되었는가? (템플릿은 별도 첨부 가능)
□ 저장 위치/파일명 규칙/전달 방법이 정해졌는가?
□ 예외가 입력/처리/출력 단계별로 최소 3개씩 정의되었는가?
□ 에스컬레이션 정책에 무응답 시 처리가 포함되었는가?
□ (새로 추가) 각 처리 단계별로 사용할 도구 범주(명령 수신기/수집기/변환기/발송기/저장기 등) 또는 실제 도구가 지정되었는가?
□ (새로 추가) 시스템 연동 시 필요한 API 키 이름이나 환경변수(.env)가 명확히 정의되었는가?
□ (새로 추가) AI 분석/요약 시 필요한 모델/엔진 종류 또는 판단 기준이 정의되었는가?
□ (새로 추가) 실행을 위해 사전에 준비되어야 할 필수 입력 파일/데이터(todo list, 프롬프트 템플릿 등)가 확보되었는가?
□ 이 워크시트를 신입사원에게 보여줘도 이해할 수 있는가?
```

**검증 방식:**
- **빈칸이 남아있는 항목** → 해당 항목을 채우기 위한 구체적 질문을 합니다.
- **내용이 작성되었으나 모호한 항목** → "이 부분은 ~라는 의미인가요?" 형태로 확인 질문을 합니다.
- **체크리스트를 모두 통과한 경우** → "자가 점검 완료"를 선언하고 보완 질문으로 넘어갑니다.

### 6-B. AI 보완 질문 (7개 관점)

AI는 작성된 워크시트를 아래 7개 관점으로 검토하고, **가장 취약한 2~3개 항목을 선별**하여 보완 질문을 합니다.

| # | 검토 항목 | 질문 예시 |
|---|---------|---------|
| 1 | 참여자 누락 | "결과물을 참조(CC)해야 할 사람이 더 있지 않나요?" |
| 2 | 수혜자 고통 누락 | "이 작업에서 실수가 발생했을 때 재작업 시간도 고통에 포함되나요?" |
| 3 | 자동화 범위 과신 | "이 단계를 🟢(완전 자동화)로 했는데, AI 판단 오류 가능성은 없나요?" |
| 4 | 입력/처리 단계 누락 | "데이터 수집 후 정제/필터링 단계가 빠져 있는 것 같은데, 별도 단계인가요?" |
| 5 | 예외 상황 미정의 | "입력 소스가 2개 이상일 때 우선순위 충돌은 어떻게 처리하나요?" |
| 6 | 에스컬레이션 현실성 | "응답 대기 1시간인데, 퇴근 후/주말에는 어떻게 하나요?" |
| 7 | 스킬 구현 가능성 | "이 플로우를 AI 스킬로 구현하려면 어떤 도구/API가 필요한가요?" |
| 8 | (새로 추가) 실행 환경 구체화 | "해당 작업을 위해 어떤 언어 모델(LLM)과 어떤 환경변수명(.env)을 사용해야 하나요?" |
| 9 | (새로 추가) 필요 데이터 사전 점검 | "작업을 돌리기 위해 필수적으로 사전에 마련되어야 하는 템플릿 파일이나 DB가 있나요?" |

### 6-C. 보완 루프 (질문 → 답변 → 수정 → 재검증)

```
[라운드 1] AI가 워크시트를 검토하고 취약한 2~3개 항목에 대해 보완 질문
    ↓
[라운드 2] 사용자가 답변
    ↓
[라운드 3] AI가 답변 내용을 워크시트에 반영하여 수정
    ↓
[라운드 4] 수정된 워크시트를 다시 체크리스트로 재검증
    ↓
[라운드 5] 미흡한 항목이 남아있으면 → [라운드 1]로 돌아가 추가 질문
         모든 항목 통과 → "워크시트 작성 완료" 선언
```

> 💡 이 루프는 **최대 3회**까지 반복합니다. 3회 후에도 미흡한 항목이 있으면 해당 항목을 명시하고, 사용자가 나중에 보완할 수 있도록 기록합니다.

### 작성 포인트
- 🔍 체크리스트 검증이 **먼저**, 보완 질문이 **다음** 순서입니다
- AI는 모든 항목을 한 번에 질문하지 않고 **반드시 하나씩 질문**하여 부담을 줄입니다
- 사용자 답변은 AI가 **워크시트에 직접 반영**하여 수정합니다
- 사용자가 현재 답할 수 없다고 밝힌 항목만 `[보류: 이유]`로 기록합니다
